ItemsView control API spec
===

# Background

Represents a control that displays a data collection.

The `ItemsView` is similar to the `ListView` and `GridView` controls, but being built using 
the `ItemsRepeater`, `ScrollView`, `ItemContainer` and `ItemCollectionTransitionProvider` components, 
it offers the unique ability to plug in custom `Layout` or `ItemCollectionTransitionProvider` 
implementations.
Another key advantage is the ability to switch the layout on the fly while preserving items selection.
The inner `ScrollView` control also offers features unavailable in the `ListView` and `GridView`  
`ScrollViewer` control like the ability to control the animation during programmatic scrolls.

Like the `ListView` and `GridView` controls, the `ItemsView` can use UI and data virtualization, 
handle keyboard, mouse, pen and touch input and has built-in accessibility support.

The inner `ScrollView` control allows scrolling and zooming of the items.


# Conceptual pages (How To)

Represents a control that displays data items.
It can organize them using various layouts like vertical stacking or columns and rows.

In the following example, the 
[`ItemsSource`](#itemsviewitemssource-property) 
property is set to a collection of photos. 
The [`ItemTemplate`](#itemsview-control-template) property specifies how each photo is represented.
Each item uses an `ItemContainer` with an `Image` child to hold the photo. 

The root element of the `ItemTemplate` template must be an `ItemContainer` otherwise
an exception is thrown.

C#
```cs
namespace PhotosApp
{
    public sealed partial class MainPage : Page
    {
        public MainPage()
        {
            this.InitializeComponent();

            this.Photos = new ObservableCollection<Photo>();
            PopulatePhotos();
        }

        public ObservableCollection<Photo> Photos
        {
            get;
            private set;
        }

        private void PopulatePhotos()
        {
            // Populates the this.Photos collection 
        }
    }
    
    public class Photo
    {
        public BitmapImage PhotoBitmapImage
        {
            get;
            set;
        }
    }
```

XAML
```xml
<Page
    x:Class="PhotosApp.MainPage"
    xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
    xmlns:local="using:PhotosApp">

    <ItemsView ItemsSource="{x:Bind Photos}" Width="120" Height="800">
        <ItemsView.ItemTemplate>
            <DataTemplate x:DataType="local:Photo">
                <ItemContainer>
                    <Image Source="{x:Bind PhotoBitmapImage, Mode=OneWay}"/>
                </ItemContainer>
            </DataTemplate>
        </ItemsView.ItemTemplate>
    </ItemsView>
</Page>
```

_Spec notes:_  
_Regarding the presence of the ItemContainer element in the Xaml markup above, we will make it 
mandatory in the first ItemsView release. We have plans to make it optional in a future release 
and also add the ability for the ItemsView to auto-generate the ItemContainer as a wrapper around
the Image element above. ItemContainer adds support for focus cues, selection cues and accessibility._
_See the future APIs named ItemsView.AreItemContainersAutoGenerated and ItemsView.GetItemContainerOverride 
in the Appendix for details._

_Also in the future APIs section of the Appendix, see the IHostItemContainer interface definition. 
For now the ItemContainer class will implement it as a cloaked interface. In the future, it will 
implement it publicly (as noted in the ItemContainer spec)._

_For the time being, the ItemsView will enforce the use of an IHostItemContainer implementation as 
the ItemTemplate template root in v1 (ItemContainer implements that interface internally).
Upon item template instantiation, the ItemsView will raise an InvalidArgument exception with a custom message.
That requirement will be lifted in the future and ItemsView.AreItemContainersAutoGenerated can then be set 
to True by default. We have run out of time to implement this feature in v1._

Following are four typical use cases of the `ItemsView` regarding the ability to select 
and/or invoke its items.

1) Items cannot be selected or invoked (read only scenarios)

2) Items can be selected but not invoked

_Spec note:_  
_c.f. Single-selectable item: TabView and NavigationView scenarios._

_c.f. Multi-selectable items: drag/drop & copy/paste purposes._

3) Items cannot be selected but can be invoked

_Spec note:_  
_c.f. Windows Start Menu scenario. Click on Notepad to launch the app._

4) Items can be selected and invoked

_Spec note:_  
_c.f. Single-selectable item: Simple email app scenario. Click on single-selectable email subject to view email in details pane. Double click it to open email in separate window._

_c.f. Multi-selectable items: File Explorer scenario. Click on file to select it and double click to open it._


Let's explore an example for each of those cases.

## Inability to select or invoke items

In this scenario, the `ItemsView` displays items that cannot be selected or invoked. Items are still 
focusable in order to support keyboard-based scrolling.

The `ItemsView`'s [`SelectionMode`](#itemsviewselectionmode-property) is set 
to `None`, disabling selections, while its 
[`IsItemInvokedEnabled`](#itemsviewisiteminvokedenabled-property) property 
keeps the default `False` value.

In this example, it is a read-only list of employee names. 

XAML
```xml
<Page
    x:Class="EmployeesApp.EmployeesOfTheMonthPage"
    xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
    xmlns:local="using:EmployeesApp">

    <ItemsView 
        ItemsSource="{x:Bind EmployeesOfTheMonth}" 
        SelectionMode="None">
        <ItemsView.Layout>
            <StackLayout Spacing="4"/>
        </ItemsView.Layout>
        <ItemsView.ItemTemplate>
            <DataTemplate x:DataType="local:Employee">
                <ItemContainer>
                    <TextBlock Text="{x:Bind FullName, Mode=OneWay}"/>
                </ItemContainer>
            </DataTemplate>
        </ItemsView.ItemTemplate>
    </ItemsView>
</Page>
```

## Ability to select items only

### Single-selectable item

In this scenario, the `ItemsView` displays navigation commands which are single-selectable. 
When an item gets selected, the `ItemsView`'s [`SelectionChanged`](#itemsviewselectionchanged-event) 
event gets raised. Application code handles that event by updating a piece 
of UI next to the `ItemsView`. The items are not invocable.

The `ItemsView`'s [`SelectionMode`](#itemsviewselectionmode-property) and 
[`IsItemInvokedEnabled`](#itemsviewisiteminvokedenabled-property) properties 
are set to their default `Single` and `False` values respectively.

An item can get selected, and thus raise the `SelectionChanged` event, programmatically 
through the `ItemsView`'s selection related methods, through UI Automation, or through user 
interaction with for example the Space key or mouse click.

XAML
```xml
<Page
    x:Class="EmployeesApp.MainPage"
    xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
    xmlns:local="using:EmployeesApp">

    <Grid ColumnDefinitions="Auto, *">
        <ItemsView 
            ItemsSource="{x:Bind AppForms}" 
            SelectionChanged="SelectAppForm" 
            Width="100">
            <ItemsView.ItemTemplate>
                <DataTemplate x:DataType="local:AppForm">
                    <ItemContainer>
                        <StackPanel Orientation="Horizontal">
                            <FontIcon Glyph="{x:Bind Glyph, Mode=OneWay}"/>
                            <TextBlock Text="{x:Bind Name, Mode=OneWay}"/>
                        </StackPanel>
                    </ItemContainer>
                </DataTemplate>
            </ItemsView.ItemTemplate>
        </ItemsView>

        <Grid x:Name="appFormGrid" Grid.Column="1"/>
    </Grid>
</Page>
```

C#
```cs
public sealed partial class MainPage : Page
{
    // ...

    private void SelectAppForm(ItemsView sender, ItemsViewSelectionChangedEventArgs e)
    {
        if (sender.SelectedItem != null)
        {
            // Hide the old AppForm UI and show the UI for the newly selected AppForm in appFormGrid.
            UpdateFormUI(sender.SelectedItem as AppForm);
        }
    }
}
```

Whereas the various AppForm instances are shown in widely different UIs in the example above,
the following example shows any selected item using the same UI. It is a list/details example 
using a binding rather than application code to automatically update the details view.

XAML
```xml
<Page

            ItemsSource="{x:Bind employeesVM}"

                <DataTemplate x:DataType="local:Employee">
                    <ItemContainer>
                        <TextBlock Text="{x:Bind FullName, Mode=OneWay}"/>
                    </ItemContainer>
                </DataTemplate>
            </ItemsView.ItemTemplate>
        </ItemsView>

        <local:EntityViewer 
            Grid.Column="1"
            Entity="{x:Bind employeesItemsView.SelectedItem, Mode=OneWay}"/>
    </Grid>

```


### Multi-selectable items

In this scenario, the `ItemsView` displays selectable items in order to be able to drag or 
copy/paste them into another application. The items are not invocable. In this case, it is a list 
of multi-selectable employee managers, laid out in rows and columns. 

The `ItemsView`'s [`SelectionMode`](#itemsviewselectionmode-property) is set to `Extended`, 
allowing multi-selections, while its [`IsItemInvokedEnabled`](#itemsviewisiteminvokedenabled-property) 
property keeps the default `False` value.

The `ItemsView`'s [`SelectionChanged`](#itemsviewselectionchanged-event) event is raised whenever 
selection changes. The event is handled to update a caption showing the total headcount of the 
selected managers' organizations.

Because the `IsItemInvokedEnabled` is `False`, the `ItemsView`'s 
[`ItemInvoked`](#itemsviewiteminvoked-event) event is never raised.

XAML
```xml
<Page
    x:Class="EmployeesApp.ManagersPage"
    xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
    xmlns:local="using:EmployeesApp">

    <Grid RowDefinitions="*, Auto">
        <ItemsView 
            ItemsSource="{x:Bind Managers}" 
            SelectionMode="Extended"
            SelectionChanged="UpdateOrgsHeadcount">
            <ItemsView.Layout>
                <UniformLayout 
                    MaximumRowsOrColumns="4"
                    MinRowSpacing="4"
                    MinColumnSpacing="4"/>
            </ItemsView.Layout>
            <ItemsView.ItemTemplate>
                <DataTemplate x:DataType="local:Employee">
                    <ItemContainer>
                        <TextBlock Text="{x:Bind FullName, Mode=OneWay}"/>
                    </ItemContainer>
                </DataTemplate>
            </ItemsView.ItemTemplate>
        </ItemsView>

        <Grid Grid.Row="1">
            <!-- Shows selected managers' organizations total headcount -->
        </Grid>
    </Grid>
</Page>
```

## Ability to invoke items only

In this scenario, the `ItemsView` is used to display a grid of application names which are focusable 
and invocable, but not selectable. When invoking the focused item, the application 
in question is launched. The `ItemsView`'s [`SelectionMode`](#itemsviewselectionmode-property) 
property is set to `None` so the items cannot be selected, and its 
[`IsItemInvokedEnabled`](#itemsviewisiteminvokedenabled-property) property is set to `True` 
so items are invocable.

The `ItemsView`'s [`ItemInvoked`](#itemsviewiteminvoked-event) event is triggered by the item's 
UI Automation, the Enter or Space keys, or Tapped event.

Because the `SelectionMode` is `None`, the `ItemsView`'s 
[`SelectionChanged`](#itemsviewselectionchanged-event) event is never raised.

XAML
```xml
<Page
    x:Class="ToolsApp.MainPage"
    xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
    xmlns:local="using:ToolsApp">

    <ItemsView 
        ItemsSource="{x:Bind ToolApplications}" 
        SelectionMode="None" 
        IsItemInvokedEnabled="True">
        <ItemsView.Layout>
            <UniformLayout 
                MaximumRowsOrColumns="4"
                MinRowSpacing="4"
                MinColumnSpacing="4"/>
        </ItemsView.Layout>
        <ItemsView.ItemTemplate>
            <DataTemplate x:DataType="local:ToolApplication">
                <ItemContainer>
                    <StackPanel Orientation="Horizontal">
                        <FontIcon Glyph="{x:Bind Glyph, Mode=OneWay}"/>
                        <TextBlock Text="{x:Bind Name, Mode=OneWay}"/>
                    </StackPanel>
                </ItemContainer>
            </DataTemplate>
        </ItemsView.ItemTemplate>
    </ItemsView>
</Page>
```

C#
```cs
public sealed partial class MainPage : Page
{
    // ...

    private void ItemsView_ItemInvoked(ItemsView sender, ItemsViewItemInvokedEventArgs e)
    {
        LaunchApplication(e.InvokedItem as ToolApplication);
    }
}
```

## Ability to select and invoke items

### Single-selectable item

In this scenario, the `ItemsView`'s [`SelectionMode`](#itemsviewselectionmode-property) 
property keeps its default `Single` value so an individual item can be selected, and its 
[`IsItemInvokedEnabled`](#itemsviewisiteminvokedenabled-property) property is set to `True` 
so items are invocable.

Both the [`SelectionChanged`](#itemsviewselectionchanged-event) and
[`ItemInvoked`](#itemsviewiteminvoked-event) events can be raised.

The `ItemInvoked` event is triggered by the item's UI Automation, the Enter key, or DoubleTapped event.

The `SelectionChanged` event is triggered programmatically through the selection related methods, 
through UI Automation, or through user interaction with the Space key or Tapped event.

In this example, the `ItemsView` presents email subjects in a vertical list, where only one email can be 
selected. On the right of the `ItemsView`, the selected email's details are shown: the sender, recipients, 
subject, body, etc...

In the `SelectionChanged` event handler, the application updates the UI showing those details for the 
`ItemsView.SelectedItem` email. 

In the `ItemInvoked` event handler, the application brings up a new window for the 
`ItemsViewItemInvokedEventArgs.InvokedItem` email. The window shows the email's details and commands to 
delete, reply, forward it.   

The user can for example select an email with a tap, and thus update the details pane on the right, and 
invoke an email with a double-tap, and thus bring it up in a separate window. 
A double-tap will first select it through the first tap if unselected, and then open it through the second tap.

XAML
```xml
<Page
    x:Class="EmailsApp.MainPage"
    xmlns="http://schemas.microsoft.com/winfx/2006/xaml/presentation"
    xmlns:x="http://schemas.microsoft.com/winfx/2006/xaml"
    xmlns:local="using:EmailsApp">

    <ItemsView 
        ItemsSource="{x:Bind Emails}" 
        IsItemInvokedEnabled="True"
        ItemInvoked="OpenEmail"
        SelectionChanged="SelectEmail">
        <ItemsView.Layout>
            <StackLayout Spacing="4"/>
        </ItemsView.Layout>
        <ItemsView.ItemTemplate>
            <DataTemplate x:DataType="local:Email">
                <ItemContainer>
                    <TextBlock Text="{x:Bind Subject, Mode=OneWay}"/>
                </ItemContainer>
            </DataTemplate>
        </ItemsView.ItemTemplate>
    </ItemsView>
</Page>
```

C#
```cs
public sealed partial class MainPage : Page
{
    // ...

    private void SelectEmail(ItemsView sender, ItemsViewSelectionChangedEventArgs e)
    {
        if (sender.SelectedItem != null)
        {
            UpdateEmailDetailsView(sender.SelectedItem);
        }
    }

    private void OpenEmail(ItemsView sender, ItemsViewItemInvokedEventArgs e)
    {
        OpenEmailInWindow(e.InvokedItem);
    }
}
```

### Multi-selectable items

In this scenario, the `ItemsView` shows items that are both selectable and invocable. As in the 
previous example items can be selected in order to copy/paste them, or drag/drop them. Items can 
also be selected in order to invoke them.

The `ItemsView`'s [`SelectionMode`](#itemsviewselectionmode-property) 
property is set to `Multiple` or `Extended` so the items can be selected, and its 
[`IsItemInvokedEnabled`](#itemsviewisiteminvokedenabled-property) property is set to `True` 
so items are invocable.

Because the `SelectionMode` is not `None`, the `ItemsView`'s 
[`SelectionChanged`](#itemsviewselectionchanged-event) event can be raised.

Because the `IsItemInvokedEnabled` is `True`, the `ItemsView`'s 
[`ItemInvoked`](#itemsviewiteminvoked-event) event can be raised. It is triggered by the item's 
UI Automation, the Enter key, or DoubleTapped event.

In this example, the `ItemsView` shows photo files. 

When the selection changed, the application's `SelectionChanged` event handler updates a caption 
showing the selected photos' date range.

When the `ItemInvoked` event is raised, the application's event handler opens all selected files 
(if there are any) in a photo editor.

XAML
```xml
<Page

    <ItemsView 
        ItemsSource="{x:Bind Photos}"
        SelectionMode="Multiple"
        SelectionChanged="UpdatePhotosDateRange"
        IsItemInvokedEnabled="True"
        ItemInvoked="EditPhotos">
        <ItemsView.Layout>
            <LinedFlowLayout LineHeight="100" LineSpacing="10" MinItemsSpacing="5"/>
        </ItemsView.Layout>
        <ItemsView.ItemTemplate>
            <DataTemplate x:DataType="local:Photo">
                <ItemContainer>
                    <Image Source="{x:Bind PhotoBitmapImage, Mode=OneWay}"/>
                </ItemContainer>
            </DataTemplate>
        </ItemsView.ItemTemplate>
    </ItemsView>
</Page>
```


## Selecting and invoking items in a nutshell

This table summarizes when the `ItemsView`'s `ItemInvoked` and `SelectionChanged` events are raised 
according to its `IsItemInvokedEnabled` and `SelectionMode` property settings:

| IsItemInvokedEnabled property | SelectionMode property | ItemInvoked event triggers | SelectionChanged event triggers |
|-|-|-|-|
| False | None | none | none |
| False | Single / Multiple / Extended | none | Programmatically via selection related methods, item's UI Automation, item's Space key, item's Tapped event |
| True | None | item's UI Automation, item's Space / Enter keys, item's Tapped event | none |
| True | Single / Multiple / Extended | item's UI Automation, item's Enter key, item's DoubleTapped event | Programmatically via selection related methods, item's UI Automation, item's Space key, item's Tapped event |

_Spec notes:_  
_Summary for the old ListView/GridView controls which have an IsItemClickEnabled property and ItemClick event:_

| _IsItemClickEnabled_ | _SelectionMode_ | _ItemClick event triggers_ |
|-|-|-|
| _False_ | _None_ / _Single_ / _Multiple_ / _Extended_ | _none_ |
| _True_ | _None_ / _Single_ / _Multiple_ / _Extended_ | _item's UIA, item's Space_ / _Enter keys, item's Tapped_ / _DoubleTapped events_ |

_ItemClickEventArgs has ClickedItem and OriginalSource properties

# API pages

## ItemsView class

Represents a control that displays data items. It can organize them using various layouts like 
vertical stacking or columns and rows.

Namespace: Microsoft.UI.Xaml.Controls

C#
```cs
public class ItemsView : Control
```

By default an `ItemsView` represents the data items in the `ItemsSource` collection in a vertical stack.


## ItemsView.ItemsSource property

Gets or sets an object source used to generate the content of the `ItemsView`.

C#
```cs
public object ItemsSource { get; set; }
```

### Property Value

The object that is used to generate the content of the inner `ItemsRepeater`. The default is `null`.

### Examples

_Spec note:_  
_ItemsView.ItemsSource is of type object because:_  
_- it is bound to the inner ItemsRepeater.ItemsSource property of type object._

### Remarks

> **Note:** For now these Remarks are copied straight from the ItemsRepeater ItemsSource help topic
https://learn.microsoft.com/en-us/windows/winui/api/microsoft.ui.xaml.controls.itemsrepeater.itemssource?view=winui-2.8
However they will be updated shortly because of inaccuracies.

Use the `ItemsSource` property to specify the collection to use to generate the content of items. You can set the `ItemsSource` 
to any type that implements `IEnumerable`. Additional collection interfaces implemented by your data source determine what 
functionality is available to the `ItemsView` to interact with your data.

This list shows available interfaces and when to consider using each one.
Other than `IKeyIndexMapping`, these interfaces provide the same behavior in `ItemsView` as they do in `ListView` and `GridView`.

* IEnumerable<T> (.NET) / IIterable<T>

    Can be used for small, static data sets.

    At a minimum, the data source must implement the IEnumerable / IIterable interface. If this is all that's supported 
    then the control will iterate through everything once to create a copy that it can use to access items via an index value.
(So there's no delay-loading perf benefit to using IEnumerable rather than IList.)

* IReadonlyList<T> (.NET) / IVectorView<T>

    Can be used for static, read-only data sets.

    Enables the control to access items by index and avoids the redundant internal copy. Data virtualization is therefore enabled.

* IList<T> (.NET) / IVector<T>

    Can be used for static data sets.

    Enables the control to access items by index and avoids the redundant internal copy. Data virtualization is therefore enabled.

    Warning: Changes to the list/vector without implementing INotifyCollectionChanged won't be reflected in the UI.

* INotifyCollectionChanged

    Recommended to support change notification.

    Enables the control to observe and react to changes in the data source and reflect those changes in the UI. 
    Projected in .NET as System.Collections.Specialized.INotifyCollectionChanged.

* IObservableVector<T>

    Supports change notification

    Like the INotifyCollectionChanged interface, this enables the control to observe and react to changes in the data source.

    Warning: The Windows.Foundation.IObservableVector<T> doesn't support a 'Move' action. This can cause the UI for an item 
    to lose its visual state. For example, an item that is currently selected and/or has focus where the move is achieved 
    by a 'Remove' followed by an 'Add' will lose focus and no longer be selected.

    The Platform.Collections.Vector<T> uses IObservableVector<T> and has this same limitation. If support for a 'Move' action 
    is required then use the INotifyCollectionChanged interface documented above. The .NET ObservableCollection<T> class uses 
    INotifyCollectionChanged.

* IKeyIndexMapping

    When a unique identifier can be associated with each item. Recommended when using 'Reset' as the collection change action.

    Enables the control to very efficiently recover the existing UI after receiving a 'Reset' action as part of 
    an INotifyCollectionChanged or IObservableVector event. After receiving a reset the control will use the provided 
    unique ID to associate the current data with elements it had already created. Without the key to index mapping 
    the control would have to assume it needs to start over from scratch in creating UI for the data.

The following interfaces on an `ItemsSource` enable special functionality in the `ListView` and `GridView` controls, 
but currently have no effect on an `ItemsRepeater` and `ItemsView`:

- `ISupportIncrementalLoading`
- `IItemsRangeInfo`
- `ISelectionInfo`

#### Comparison with ItemsControl.ItemsSource

`ItemsView` builds on `ItemsRepeater`, so its supported `ItemsSource` interfaces differ from the legacy `ItemsControl.ItemsSource` (used by `ListView`, `GridView`, `ListBox`, etc.). Do not assume full parity:

* No `Items` collection: unlike `ItemsControl`, `ItemsView` has no `Items` property — data must be set via `ItemsSource`.
* Honored by `ListView`/`GridView` but ignored by `ItemsView`/`ItemsRepeater`: `ISupportIncrementalLoading`, `IItemsRangeInfo`, `ISelectionInfo`.
* Same behavior: `IEnumerable`/`IIterable`, `IReadOnlyList`/`IVectorView`, `IList`/`IVector`, `INotifyCollectionChanged`, `IObservableVector`, and `IKeyIndexMapping` (consumed by `ItemsView` via `ItemsRepeater` like `ListViewBase`; only the bare `ItemsControl` ignores it).
* Minimum requirement is the same as `ItemsControl`: the source must implement `IEnumerable`/`IIterable<T>`.

Gets or sets the template used to display each item.

C#


### Property Value

The template that specifies the visualization of the data objects. The default is `null`.

### Remarks

When set to a non-null value, the `ItemTemplate` property must be such that once instantiated, 
the template's root element must be an `ItemContainer`.

_Spec notes:_  
_In fact the template's root element must implement the internal `IHostItemContainer` interface.

to display the item at index 0 might be reused later to display the item at index 100.

may be assigned a new `Source` to represent another item.

A user can toggle the selection of a single item by using the space bar, mouse click, or touch tap.

Gets the loaded `ScrollView` control template part.


### Property Value

it was loaded. 

Returns `null` before the control template gets loaded or when the part is missing.

_Spec note:_  
_There are pros and cons of exposing a template part through the public api. It results in a very flexible design, but
it can result in some undesired behavior if the app author modifies the template part in unexpected ways. For example,
if the app author were to change the ScrollView.Content property to something else, the ItemsView would stop functioning._
_The alternative to exposing the template part as a public api is to either mirror all of the relevant properties of 
ScrollView onto ItemsView or to expose some kind of proxy object in lieu of the ScrollView. Both of these approaches are 
a lot more complex, so it was decided that this was a reaonable trade-off._
_In general, ItemsView will work well as long as the app author does not do anything 'unreasonable' to the ScrollView. 
(We don't try to define explicitly what is reasonable or unreasonable.)_ 
_If we chose not to expose this property, it would not prevent an app author from getting access to this object anyway
since and app author can always find the template parts with VisualTreeHelper. So it was decided that the convenience 
that this property provides was worth the tradeoffs._

### Property Value

The default is `null`. After the `ItemsView` control template is loaded, this property returns

### Remark

Changing the `ItemsView.VerticalScrollController` property to a custom value after control template

of also restoring the original `ScrollView.VerticalScrollBarVisibility` value.
That is, if the `VerticalScrollBarVisibility` was automatically changed from `Visible` to `Hidden` 
during the custom `IScrollController` use, it is restored to `Visible` when the built-in scrollbar 
is used again.

vertical scrollbar, and a `LinedFlowLayout` with an `AnnotatedScrollBar`.

## ItemsView.TryGetItemIndex method

Retrieves the index of the closest item to a position relative to the `ItemsView` top/left corner.

C#
```cs
public bool TryGetItemIndex(double horizontalViewportRatio, double verticalViewportRatio, out int index);
```

### Remarks

Returns `False` when the control's inner `ItemsRepeater` or `ScrollView` part is not set, or the data source 
`ItemsSource` is empty. The index out parameter is set to -1 in those cases.

Otherwise, the index is set to a value greater or equal to 0 and returns `True`.

The `TryGetItemIndex` method can be called in an `ItemsView.Loaded` event handler or any later time 
so that the inner control template parts are loaded.

The input parameters are interpreted as follows:

| **Argument value** | **Effect** |
|-|-|
| horizontalViewportRatio <= 0.5 | Return item with left edge closest to viewport ratio point |
| horizontalViewportRatio > 0.5  | Return item with right edge closest to viewport ratio point |
| verticalViewportRatio <= 0.5   | Return item with top edge closest to viewport ratio point    |
| verticalViewportRatio > 0.5    | Return item with bottom edge closest to viewport ratio point |

_Spec note:_  
_The LinedFlowLayout needs to go through a full Measure/Arrange pass before the TryGetItemIndex call. 
Layout occurs before the Loaded event is raised, so this method can be safely called from a Loaded event 
handler._


When the `ItemsView`'s `FlowDirection` is `RightToLeft`, the top/right corner is the reference point.
When `horizontalViewportRatio` <= 0.5, item with right edge closest to viewport ratio point is returned.
When `horizontalViewportRatio` > 0.5, item with left edge closest to viewport ratio point is returned.

_Spec note:_  
_Layout.IndexBasedLayoutOrientation is a new Layout property that is described in the LinedFlowLayout and Layout API spec._

It's possible for there to be different matching elements for the vertical or horizontal ratio. 
This is resolved based on the `Layout`'s `IndexBasedLayoutOrientation` property.

| **Layout.IndexBasedLayoutOrientation value** | **Effect** |
|-|-|
| IndexBasedLayoutOrientation.Horizontal | Vertical distances take precedence over horizontal distances, i.e. the horizontal distances are ignored unless the vertical distances are equal. |
| IndexBasedLayoutOrientation.Vertical | Horizontal distances take precedence over vertical distances, i.e. the vertical distances are ignored unless the horizontal distances are equal. |
| IndexBasedLayoutOrientation.None | Horizontal and vertical distances have equal importance. |


These examples use a `Layout.IndexBasedLayoutOrientation` returning `IndexBasedLayoutOrientation.Horizontal`.

Red rectangles represent viewports. A red dot is a "viewport ratio point", while a black dot represents the returned 
item's point used to evaluate the distance to the viewport ratio point.

| **`horizontalViewportRatio` value** | **`verticalViewportRatio` value** | **Returned value** | **Illustration** |
|-|-|-|-|
|  0.0 |  0.0 |  7 | ![Illustration of ItemsView's TryGetItemIndex method with arguments (0.0, 0.0)](Images/TryGetItemIndexScenario1.png) |
|  1.0 |  0.0 | 12 | ![Illustration of ItemsView's TryGetItemIndex method with arguments (1.0, 0.0)](Images/TryGetItemIndexScenario2.png) |
|  1.0 |  1.0 | 21 | ![Illustration of ItemsView's TryGetItemIndex method with arguments (1.0, 1.0)](Images/TryGetItemIndexScenario3.png) |
|  0.0 |  1.0 | 19 | ![Illustration of ItemsView's TryGetItemIndex method with arguments (0.0, 1.0)](Images/TryGetItemIndexScenario4.png) |
|  1.0 |  2.0 | 25 | ![Illustration of ItemsView's TryGetItemIndex method with arguments (1.0, 2.0)](Images/TryGetItemIndexScenario5.png) |
| -1.0 | -1.0 | 16 | ![Illustration of ItemsView's TryGetItemIndex method with arguments (-1.0, -1.0)](Images/TryGetItemIndexScenario6.png) |

Because the `Layout.IndexBasedLayoutOrientation` is `Horizontal`, TryGetItemIndex(1.0, 1.0, out index) 
returns index 32 instead 29 in the following example, even though the bottom/right corner of 
item 29 is closer to the bottom/right corner of the red viewport.

![Illustration of Layout.IndexBasedLayoutOrientation's effect on ItemsView.TryGetItemIndex](Images/TryGetItemIndexScenario7.png)

The concept of `Layout.IndexBasedLayoutOrientation` also exists so that when item 32 has keyboard focus, 
hitting the right arrow key has no effect even though item 29 is on its right.

### Examples

#### Simple example

The developer wants to determine the approximate range of visible items in the `ItemsView`. 
TryGetItemIndex is called to the top/left and bottom/right corners of the `ItemsView`.

C#
```cs
public sealed partial class MainPage : Page
{
    // ...

    private void GetItemsRange(ref int indexCloseToTopLeftCorner, ref int indexCloseToBottomRightCorner)
    {
        myItemsView.TryGetItemIndex(horizontalViewportRatio: 0.0, verticalViewportRatio: 0, out indexCloseToTopLeftCorner);
        myItemsView.TryGetItemIndex(horizontalViewportRatio: 1.0, verticalViewportRatio: 1.0, out indexCloseToBottomRightCorner);
    }
}
```


#### Advanced example

The developer wants to update a label based on the first item at the top of the viewport with at least 10 visible 
pixels vertically, and the last item at the bottom of the viewport with at least 10 visible pixels.

![Illustration of ItemsView's TryGetItemIndex method advanced scenario](Images/FirstAndLastRelevantItems.png)

C#
```cs
public sealed partial class MainPage : Page
{
    // ...

    private void UpdateDateRangeLabel()
    {
        const double requiredDisplayedHeight = 10.0; // Ignore items which do not show at least 10 pixels in height.

        LinedFlowLayout linedFlowLayout = myItemsView.Layout as LinedFlowLayout;

        int firstRelevantItemIndex, lastRelevantItemIndex;
        
        myItemsView.TryGetItemIndex(
            horizontalViewportRatio: 0.0, 
            verticalViewportRatio: ((linedFlowLayout.LineSpacing - linedFlowLayout.LineHeight) / 2.0 + requiredDisplayedHeight) / myItemsView.ActualHeight,
            out firstRelevantItemIndex);
        myItemsView.TryGetItemIndex(
            horizontalViewportRatio: 1.0, 
            verticalViewportRatio: 1.0 + ((linedFlowLayout.LineHeight + linedFlowLayout.LineSpacing) / 2.0 - requiredDisplayedHeight) / myItemsView.ActualHeight,
            out lastRelevantItemIndex);

        string firstRelevantItemDate = (myItemsView.ItemsSource[firstRelevantItemIndex] as Photo).Date.ToString();
        string lastRelevantItemDate = (myItemsView.ItemsSource[lastRelevantItemIndex] as Photo).Date.ToString();

        myRangeLabel.Text = firstRelevantItemDate + " - " + lastRelevantItemDate;
    }
}
```

`firstRelevantItemDate` is evaluated to 10 as it is the first visible item showing at least 10px vertically.
`lastRelevantItemDate` is evaludate to 23. Item 25 does not qualify since it shows less than 10px vertically 
and it considered irrelevant by the app developer.

_Spec notes:_  
_"What does Layout.IndexBasedLayoutOrientation property do?" Nothing. It is being used by ItemsView.TryGetItemIndex 
and also ItemsView internal code to process keyboard-based navigation.
For example, LinedFlowLayout returns Horizontal. The Right-Arrow handling is thus navigating to the next index
(moving from index I to index I+1).
Likewise, the Left-Arrow handling is thus navigating to the previous index (moving from index I to index I-1).
Left-Arrow and Right-Arrow are moving to an item based on its index.
Up-Arrow and Down-Arrow are moving to an item based on its physical position.
The behaviors are reversed when Layout.IndexBasedLayoutOrientation returns Vertical.
When Layout.IndexBasedLayoutOrientation returns None, all four arrows move to an item based on its physical location, rather than index._

## ItemsView.StartBringItemIntoView method

Starts an operation to bring the item with the specified index into view.

C#
```cs
public void StartBringItemIntoView(int index, Microsoft.UI.Xaml.BringIntoViewOptions options);
```

### Remarks

The `StartBringIntoView` method is internally invoked for the `UIElement` associated with the index.
If that UI element is currently virtualized, it is first realized and synchronously laid out, prior to 
the `StartBringIntoView` call.

The `options` argument can be `null` and `UIElement.StartBringIntoView()` is invoked in that case.
When non-null, the `UIElement.StartBringIntoView(BringIntoViewOptions options)` method is invoked.

Animated operations to virtualized items are not supported - the inner `ScrollView` will not perform an 
animated scroll when the target first needs to be realized and laid out.

For the `StartBringItemIntoView` method to be successfully executed, 
- the `ItemsView.ItemsSource` property must not be `null`, 
- the provided index must be and remain valid given the `ItemsView.ItemsSource` collection,
- an `ItemsRepeater` must have been loaded as part of the `ItemsView`'s control template,
- and the `ItemsView` must be parented.

_Spec notes:_  
_An invalid-operation exception is thrown when:
- the `ItemsView.ItemsSource` property is `null`,
- the `ItemsView` control template was not loaded, and thus the `ItemsView.Loaded` event was not raised yet,
- or there is no inner `ItemsRepeater` in the loaded `ItemsView` control template._

_An out-of-bounds exception is thrown when:
- the provided index is not valid for the `ItemsView.ItemsSource` collection at the beginning of the 
`StartBringItemIntoView` call,
- the provided index is no longer valid after attempting to realize and laying out the virtualized target element._

_When throwing such an exception, a helpful error message will be provided through the use of 
[RoOriginateError](https://learn.microsoft.com/en-us/windows/win32/api/roerrorapi/nf-roerrorapi-rooriginateerror)._
_That capability may have to be added to MuxC as it may only be available in Mux at the moment. This would be 
treated as a separate Task._

### Example

After an application start-up, the developer wants to bring the item that was at the top of the `ItemsView` 
prior to the application shutdown back to the very top.

The `StartBringItemIntoView` call is made in an `ItemsView.Loaded` event handler to guarantee that 
the control is ready to perform the request.

C#
```cs
public sealed partial class MainPage : Page
{
    private int m_topItemIndexToRestore;

    public MainPage()
    {
        this.InitializeComponent();

        m_topItemIndexToRestore = RetrieveTopItemIndexToRestore();

        myItemsView.Loaded += MyItemsView_Loaded;
    }

    private void MyItemsView_Loaded(object sender, RoutedEventArgs e)
    {
        BringIntoViewOptions options = new BringIntoViewOptions() { VerticalAlignmentRatio = 0.0 };

        myItemsView.StartBringItemIntoView(m_topItemIndexToRestore, options);
    }
}
```

_Spec notes:_  
_During the initial realization and layout phases of an ItemsView.StartBringItemIntoView call, the ItemsRepeater 
may raise a bunch of [ItemsRepeater.ElementClearing](https://docs.microsoft.com/windows/windows-app-sdk/api/winrt/Microsoft.UI.Xaml.Controls.ItemsRepeater.ElementClearing) 
and [ItemsRepeater.ElementPrepared](https://docs.microsoft.com/windows/windows-app-sdk/api/winrt/Microsoft.UI.Xaml.Controls.ItemsRepeater.ElementPrepared) 
events, and the LinedFlowLayout (if used) raises its ItemsInfoRequested event._

ItemsView supports bottom-up lists (i.e. a chat scenario) by setting the inner `ScrollView`'s `VerticalAnchorRatio` to 1.0.
For a bring-into-view operation, `BringIntoViewOptions.VerticalAlignmentRatio` can be set to 1.0 to bring an item to 
the bottom of the viewport.


## ItemsView.ItemInvoked event

Occurs when an item in the `ItemsView` control receives a user interaction, while the [`IsItemInvokedEnabled`](#itemsviewisiteminvokedenabled-property) 
property is set to `True`.

C#
```cs
public event TypedEventHandler<ItemsView, ItemsViewItemInvokedEventArgs> ItemInvoked;
```

### Remark

This event is not raised when the `IsItemInvokedEnabled` is set to `False`. Otherwise, it is raised in any of these five cases: 

- an item is invoked through UI Automation,
- an item received an Enter keystroke,
- the `SelectionMode` property is `None` and an item received a Space keystroke,
- the `SelectionMode` property is `None` and an item raised its Tapped event,
- the `SelectionMode` property is different from `None` and an item raised its DoubleTapped event.

## ItemsView.SelectionChanged event

Occurs whenever selection changes (SelectedItem and SelectedItems properties have changed). This can for example be due

C#
```cs

```

### Remark

This event is raised for example when an item is selected through UI Automation, or when it

_Spec note:_  
_The ItemsViewSelectionChangedEventArgs class currently exposes no API because SelectionModelSelectionChangedEventArgs

```xml
            ItemsSource="{x:Bind AppForms}"
```

## Other component APIs created for ItemsView

### ScrollingScrollMode.Auto enum value

The old `ScrollViewer` control uses a [`ScrollMode`](https://learn.microsoft.com/en-us/uwp/api/windows.ui.xaml.controls.scrollmode?view=winrt-22621) 
enum for its `HorizontalScrollMode` and `VerticalScrollMode` properties.

It is defined as:

| **Name** | **Value** | **Description** |
|-|-|-|
| Disabled | 0 | Scrolling is disabled. |
| Enabled | 1 | Scrolling is enabled. |
| Auto | 2 | Scrolling is enabled only when the content extent is larger than the viewport size. |

When we originally defined the new `ScrollView` / `ScrollPresenter` public APIs we did not have a 
compelling scenario for `ScrollingScrollMode.Auto` and only kept `ScrollingScrollMode.Enabled` / 
`ScrollingScrollMode.Disabled` leaving the door open for adding an `Auto` enum value in the future. 
The `Auto` support was actually implemented but commented out.

Since then we have received feedback from `ItemsView` users pointing to the need for the `ScrollingScrollMode.Auto` 
value. The `Auto` support was actually implemented but commented out.

In the `ScrollViewer` case, when the content is smaller than the viewport, 
- `Enabled` lets the user overpan the content with touch (even though it is small and the scroll offset sticks to 0.0),
- `Auto` does not let the user overpan the content with touch when it's smaller than the viewport.

That behavior will be replicated in the new `ScrollView` control.

The feedback with the `ItemsView` used in the File Explorer Photos Gallery is that while the images fit within the 
width of the `ItemsView`, the user can overpan them horizontally with touch. This is because the `Auto` enum value 
is not available, and the `ScrollView.HorizontalScrollMode` property is set to `Enabled` to accommodate 
`LinedFlowLayout` lines that exceed the width of the `ItemsView`.

So the plan is:
- add the `ScrollingScrollMode.Auto` enum,
- add the `ScrollView` and `ScrollPresenter` `ComputedHorizontalScrollMode` and `ComputedVerticalScrollMode` read-only 
  dependency properties,
- uncomment the `ScrollPresenter` / `ScrollView` code supporting those APIs,
- have File Explorer code set `ScrollView.HorizontalScrollMode` to `Auto`.

The enum will then be defined as:

C#
```cs
    enum ScrollingScrollMode
    {
        Enabled = 0,
        Disabled = 1,
        Auto = 2      // New value only
    };
```


### ScrollView.ComputedHorizontalScrollMode / ScrollView.ComputedVerticalScrollMode / ScrollPresenter.ComputedHorizontalScrollMode / ScrollPresenter.ComputedVerticalScrollMode properties

```cs
public ScrollingScrollMode ComputedHorizontalScrollMode { get; }
```

### Property Value

`ComputedHorizontalScrollMode` returns the same value as `HorizontalScrollMode` when it is either

If the template root already implements `IHostItemContainer` or the `GetItemContainerOverride` method returns `null`, 
the template root is used as is.

If the `GetItemContainerOverride` method returns a `IHostItemContainer` instance, it is used as a wrapper for 
the template root. The `ItemsView` automatically invokes `IHostItemContainer.SetChild(UIElement)` using the template 
root as the argument. The `ItemContainer` implementation of `SetChild` simply sets the `ItemContainer.Child` property. 
If a custom `IHostItemContainer` implementation were a `ContentControl`, it would set its `Content` instead.

The `ItemsRepeater` finally uses either the unwrapped template root or the `IHostItemContainer` instance as 
the `ItemsRepeaterElementPreparedEventArgs.Element` property value when raising its `ElementPrepared` event.

The `ItemsView` class uses `ItemContainer` as its default item container type. Its implementation of 
`GetItemContainerOverride` simply returns such a new `ItemContainer`.
Derived classes of `ItemsView` can use their own custom type of containers instead.

#### Example

This example overrides the default behavior by setting the `AreItemContainersAutoGenerated` property to `False`. 
This means that the `Image` element is kept as the root element for each item instead of being wrapped into 
an `ItemContainer`. The applied `SelectionMode` being `None`, the `ItemContainer` wrappers would have little benefit. 
The `ItemContainer`, through its `IHostItemContainer` implementation, indeed shows visual cues for selection and raises 
user interaction events which are not needed when selection is turned off at the `ItemsView` level.

```xml
        SelectionMode="None"
```

## ScrollView.ContentOrientation attached dependency property

The `ScrollView` control has a `ContentOrientation` that affects the available size provided to its 
content. Its default value is `Vertical`.

When the `ContentOrientation` is `Vertical`, the `ScrollView`'s content is given an available width 
that matches the `ScrollView`'s width, and is given an infinite available height, favoring vertical 
scrolling capability.
On the other hand, when the `ContentOrientation` is `Horizontal`, the `ScrollView`'s content is given 
an available height that matches the `ScrollView`'s height, and is given an infinite available width, 
favoring horizontal scrolling capability.

_Spec note:_  
_`ScrollView.ContentOrientation` indicates the preferred content orientation. The `ContentOrientation` 
property affects the `ScrollPresenter`'s MeasureOverride behavior:_

_When the Vertical orientation is applied, the ScrollPresenter.Content's available width is set to the 
ScrollPresenter's available width. Otherwise, the ScrollPresenter.Content's available width is infinity._

_When the Horizontal orientation is applied, the ScrollPresenter.Content's available height is set to the 
ScrollPresenter's available height. Otherwise, the ScrollPresenter.Content's available height is infinity._

### GetContentOrientation method

## Example of custom IHostItemContainer implementation

Following is a custom implementation of IHostItemContainer, a custom item container that 
can be used with the `ItemsView`.

C#
```cs
using Windows.Foundation;
using Microsoft.UI.Xaml;
using Microsoft.UI.Xaml.Automation.Peers;
using Microsoft.UI.Xaml.Automation.Provider;
using Microsoft.UI.Xaml.Controls;
using Microsoft.UI.Xaml.Input;

namespace CustomControls
{
    public class CustomItemContainer : ContentControl, IHostItemContainer
    {
        private class PointerInfo
        {
            public bool IsPressed { get; set; }
            public bool IsPointerOver { get; set; }

            // Keeps track of pointer information
        }

        private PointerInfo _pointerInfo = new PointerInfo();
        private CheckBox _selectionCheckBox = null;

        // IHostItemContainer-related fields
        private HostItemContainerSelectionMode _selectionMode = HostItemContainerSelectionMode.Single;
        private bool _hostCanUserInvoke, _hostCanUserSelect;
        private HostItemContainerItemInvokedEventArgs _hostItemContainerItemInvokedEventArgs = new HostItemContainerItemInvokedEventArgs();
        private HostItemContainerIsSelectedChangedEventArgs _hostItemContainerIsSelectedChangedEventArgs = new HostItemContainerIsSelectedChangedEventArgs();
        private event TypedEventHandler<IHostItemContainer, HostItemContainerItemInvokedEventArgs> _itemInvoked;
        private event TypedEventHandler<IHostItemContainer, HostItemContainerIsSelectedChangedEventArgs> _isSelectedChanged;

        public static readonly DependencyProperty CanUserInvokeProperty = DependencyProperty.Register(
          "CanUserInvoke",
          typeof(bool),
          typeof(CustomItemContainer),
          new PropertyMetadata(true, null));

        public static readonly DependencyProperty CanUserSelectProperty = DependencyProperty.Register(
          "CanUserSelect",
          typeof(bool),
          typeof(CustomItemContainer),
          new PropertyMetadata(true, null));

        public static readonly DependencyProperty IsSelectedProperty = DependencyProperty.Register(
          "IsSelected",
          typeof(bool),
          typeof(CustomItemContainer),
          new PropertyMetadata(false, new PropertyChangedCallback(OnIsSelectedChanged)));

        public CustomItemContainer() 
        {
            IsEnabledChanged += OnIsEnabledChanged;
        }

        #region IHostItemContainer implementation

        event TypedEventHandler<IHostItemContainer, HostItemContainerItemInvokedEventArgs> IHostItemContainer.ItemInvoked
        {
            add
            {
                _itemInvoked += value;
            }

            remove
            {
                _itemInvoked -= value;
            }
        }

        event TypedEventHandler<IHostItemContainer, HostItemContainerIsSelectedChangedEventArgs> IHostItemContainer.IsSelectedChanged
        {
            add
            {
                _isSelectedChanged += value;
            }

            remove
            {
                _isSelectedChanged -= value;
            }
        }

        HostItemContainerSelectionMode IHostItemContainer.SelectionMode
        {
            get
            {
                return _selectionMode;
            }

            set
            {
                if (value != _selectionMode)
                {
                    _selectionMode = value;

                    UpdateVisualState(true);
                    UpdateMultiSelectState(true);
                }
            }
        }
        
        UIElement IHostItemContainer.Child 
        { 
            get
            {
                return Content as UIElement;
            }

            set
            {
                Content = value;
            }
        }

        bool IHostItemContainer.IsSelected 
        { 
            get
            {
                return IsSelected;
            }

            set
            {
                IsSelected = value;
            }
        }

        bool IHostItemContainer.GetEffectiveCanUserInvoke()
        {
            return _hostCanUserInvoke && CanUserInvoke;
        }

        bool IHostItemContainer.GetEffectiveCanUserSelect()
        {
            return _hostCanUserSelect && CanUserSelect;
        }

        void IHostItemContainer.SetCanUserInvoke(bool value)
        {
            _hostCanUserInvoke = value;
        }

        void IHostItemContainer.SetCanUserSelect(bool value)
        {
            _hostCanUserSelect = value;
        }

        #endregion

        public bool CanUserInvoke
        {
            get
            {
                return (bool)GetValue(CanUserInvokeProperty);
            }

            set
            {
                SetValue(CanUserInvokeProperty, value);
            }
        }

        public bool CanUserSelect
        {
            get
            {
                return (bool)GetValue(CanUserSelectProperty);
            }

            set
            {
                SetValue(CanUserSelectProperty, value);
            }
        }

        public bool IsSelected
        {
            get
            {
                return (bool)GetValue(IsSelectedProperty);
            }

            set
            {
                SetValue(IsSelectedProperty, value);
            }
        }

        protected override void OnApplyTemplate()
        {
            _selectionCheckBox = null;

            base.OnApplyTemplate();

            // set _selectionCheckBox and register its Checked & Unchecked events

            UpdateVisualState(true);
            UpdateMultiSelectState(true);
        }

        protected override AutomationPeer OnCreateAutomationPeer()
        {
            // Because CustomItemContainer uses a CustomItemContainerAutomationPeer instead of 
            // ItemContainerAutomationPeer, an ItemInvoked event is needed on IHostItemContainer.
            // Without that event, UIA could not trigger the ItemsView.ItemInvoked event.
            return new CustomItemContainerAutomationPeer(this);

            // If CustomItemContainer were to reuse ItemContainerAutomationPeer, no such event 
            // would be needed:
            // return new ItemContainerAutomationPeer(this);
            // This code 
            // ItemContainerAutomationPeer::ItemContainerAutomationPeer(winrt::ItemContainer const& owner) : ReferenceTracker(owner) {}
            // would need to be changed to
            // ItemContainerAutomationPeer::ItemContainerAutomationPeer(winrt::FrameworkElement const& owner) : ReferenceTracker(owner) { // make sure owner implements IHostItemContainer }
            // and ItemContainerAutomationPeer's "auto itemContainer = Owner().try_as<winrt::ItemContainer>()" needs to change to
            // "auto hostItemContainer = Owner().try_as<winrt::IHostItemContainer>()"
            
            // Is it acceptable to force IHostItemContainer custom implementations to use 
            // ItemContainerAutomationPeer?
        }

        internal void RaiseItemInvoked()
        {
            if (_itemInvoked != null)
            {
                _itemInvoked(this, _hostItemContainerItemInvokedEventArgs);
            }
        }

        private static void OnIsSelectedChanged(DependencyObject d, DependencyPropertyChangedEventArgs e)
        {
            ((CustomItemContainer)d).OnIsSelectedChanged();
        }

        private void OnIsSelectedChanged()
        {
            UpdateVisualState(true);
            UpdateMultiSelectState(true);

            RaiseSelectionAutomationEvent();
            RaiseIsSelectedChanged();
        }

        private void OnIsEnabledChanged(object sender, DependencyPropertyChangedEventArgs e)
        {
            if (!IsEnabled)
            {
                // update _pointerInfo
            }

            UpdateVisualState(true);
        }

        private void RaiseSelectionAutomationEvent()
        {
            bool isMultipleOrExtended = _selectionMode == HostItemContainerSelectionMode.Multiple || _selectionMode == HostItemContainerSelectionMode.Extended;

            AutomationEvents eventToRaise =
                IsSelected ?
                (isMultipleOrExtended ? AutomationEvents.SelectionItemPatternOnElementAddedToSelection : AutomationEvents.SelectionItemPatternOnElementSelected) :
                (isMultipleOrExtended ? AutomationEvents.SelectionItemPatternOnElementRemovedFromSelection : AutomationEvents.PropertyChanged);

            if (eventToRaise != AutomationEvents.PropertyChanged && AutomationPeer.ListenerExists(eventToRaise))
            {
                AutomationPeer automationPeer = FrameworkElementAutomationPeer.CreatePeerForElement(this);

                if (automationPeer != null)
                {
                    automationPeer.RaiseAutomationEvent(eventToRaise);
                }
            }
        }

        private void RaiseIsSelectedChanged()
        {
            if (_isSelectedChanged != null)
            {
                _isSelectedChanged(this, _hostItemContainerIsSelectedChangedEventArgs);
            }
        }

        private void UpdateVisualState(bool useTransitions)
        {
            // DisabledStates
            if (!IsEnabled)
            {
                VisualStateManager.GoToState(this, "Disabled", useTransitions);
            }
            else
            {
                VisualStateManager.GoToState(this, "Enabled", useTransitions);

                // CombinedStates
                if (_pointerInfo.IsPressed && IsSelected)
                {
                    VisualStateManager.GoToState(this, "SelectedPressed", useTransitions);
                }
                else if (_pointerInfo.IsPressed)
                {
                    VisualStateManager.GoToState(this, "UnselectedPressed", useTransitions);
                }
                else if (_pointerInfo.IsPointerOver && IsSelected)
                {
                    VisualStateManager.GoToState(this, "SelectedPointerOver", useTransitions);
                }
                else if (_pointerInfo.IsPointerOver)
                {
                    VisualStateManager.GoToState(this, "UnselectedPointerOver", useTransitions);
                }
                else if (IsSelected)
                {
                    VisualStateManager.GoToState(this, "SelectedNormal", useTransitions);
                }
                else
                {
                    VisualStateManager.GoToState(this, "UnselectedNormal", useTransitions);
                }
            }
        }

        private void UpdateMultiSelectState(bool useTransitions)
        {
            // MultiSelectStates
            if (_selectionMode == HostItemContainerSelectionMode.Multiple)
            {
                VisualStateManager.GoToState(this, "Multiple", useTransitions);

                UpdateCheckboxState();
            }
            else
            {
                VisualStateManager.GoToState(this, "Single", useTransitions);
            }
        }

        private void UpdateCheckboxState()
        {
            if (_selectionCheckBox != null && (bool)_selectionCheckBox.IsChecked != IsSelected)
            {
                _selectionCheckBox.IsChecked = IsSelected;
            }
        }

        protected override void OnPointerEntered(PointerRoutedEventArgs args)
        {
            base.OnPointerEntered(args);

            // updates _pointerInfo, may call UpdateVisualState(true)
        }

        protected override void OnPointerMoved(PointerRoutedEventArgs args)
        {
            base.OnPointerMoved(args);

            // updates _pointerInfo, may call UpdateVisualState(true)
        }

        protected override void OnPointerExited(PointerRoutedEventArgs args)
        {
            base.OnPointerMoved(args);

        }

        protected override void OnPointerPressed(PointerRoutedEventArgs args)
        {
            base.OnPointerPressed(args);

            // updates _pointerInfo, may call UpdateVisualState(true)
        }

        protected override void OnPointerReleased(PointerRoutedEventArgs args)
        {
            base.OnPointerReleased(args);

            // updates _pointerInfo, may call UpdateVisualState(true)
        }

        protected override void OnPointerCanceled(PointerRoutedEventArgs args)
        {
            base.OnPointerCanceled(args);

            // update _pointerInfo, may call UpdateVisualState(true);
        }

        protected override void OnPointerCaptureLost(PointerRoutedEventArgs args)
        {
            base.OnPointerCaptureLost(args);

            // update _pointerInfo, may call UpdateVisualState(true);
        }
    }
```
